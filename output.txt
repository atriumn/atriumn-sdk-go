Event Patterns\n\n### 2.1 Events Published\n\nThe SDK does not directly publish events to external systems. Event publishing is handled by the backend services.\n\n### 2.2 Events Consumed\n\nThe SDK does not directly consume events from external systems. Instead, it polls for status changes or receives webhooks through API endpoints.\n\n#### Asynchronous Processing Status Updates\n\nWhile not technically events, the SDK handles asynchronous processing status updates for:\n\n1. **URL Ingestion Processing**\n   - URL content is processed asynchronously\n   - Initial status is `PENDING` or `PROCESSING`\n   - Client application must poll for status changes\n\n2. **File Processing**\n   - Large files may undergo processing after upload\n   - Initial status is `UPLOADING` or `PROCESSING`\n   - Final status becomes `COMPLETED` or `ERROR`\n\n### 2.3 Message Formats\n\nNo direct message consumption occurs within the SDK. Status updates are delivered through standard API responses.\n\n### 2.4 Delivery Guarantees\n\nThe SDK relies on HTTP response codes and retries for reliable communication:\n\n- Automatic retries for network errors (configurable in client options)\n- No guaranteed delivery for asynchronous operations\n- Client applications should implement their retry logic for status polling\n\n## 3. Service Dependencies\n\n### 3.1 External Services\n\nThe Atriumn SDK for Go depends on the following external services:\n\n1. **Atriumn Auth Service**\n   - Provides authentication and user management\n   - Used for direct file uploads and downloads\n   - Accessed via pre-signed URLs\n\n### 3.2 Required Capabilities\n\nRequired capabilities from external services:\n\n1. **Auth Service**\n   - OAuth 2.0 client credentials flow\n   - JWT token issuance and validation\n   - User management (signup, login, password reset)\n\n2. **AI Service**\n   - Prompt template storage and retrieval\n   - Prompt versioning\n   - Tagging and filtering\n\n3. **Storage Service**\n   - Generation of S3 pre-signed URLs\n   - Access control based on tenant ID\n   - File metadata management\n\n4. **Ingest Service**\n   - Content ingestion and processing\n   - Content indexing and retrieval\n   - Timeouts default to 10 seconds (configurable)\n   - Network errors are converted to `apierror.ErrorResponse` with code `network_error`\n\n2. **Authentication Failures**\n   - 401 responses are converted to `apierror.ErrorResponse` with code `unauthorized`\n   - 403 responses are converted to `apierror.ErrorResponse` with code `forbidden`\n\n3. **Service Unavailability**\n   - 5xx responses are converted to `apierror.ErrorResponse` with code `server_error`\n   - Default behavior does not include automatic retries, but clients can implement this\n\n4. **Rate Limiting**\n   - 429 responses are converted to `apierror.ErrorResponse` with code `rate_limited`\n   - No built-in backoff, client applications should implement\n\n5. **Context Cancellation**\n   - All API calls accept a context.Context parameter\n   - Allows client applications to implement timeouts and cancellation\n\n## 4. Data Exchange\n\n### 4.1 Shared Data Models\n\nThe SDK defines Go struct representations of the following shared data models:\n\n1. **Authentication Models**\n   - `TokenResponse`: Contains access token, token type, and expiration\n   - `ClientCredentialResponse`: Client credential details\n   - `UserProfileResponse`: User profile information\n\n2. **AI Prompt Models**\n   - `Prompt`: Complete prompt configuration\n   - `PromptVariable`: Variable definition for prompt templates\n\n3. **Storage Models**\n   - `GenerateUploadURLResponse`: Pre-signed URL for uploads\n   - `GenerateDownloadURLResponse`: Pre-signed URL for downloads\n\n4. **Ingest Models**\n   - `ContentItem`: Metadata about ingested content\n   - Used for all HTTP request and response bodies\n   - Implemented using Go's `encoding/json` package\n   - All structs have appropriate json tags\n\n2. **HTTP Headers**\n   - Standard HTTP headers for authentication and content negotiation\n   - Content-Type: application/json\n   - Accept: application/json\n   - Authorization: Bearer {token}\n\n3. **Query Parameters**\n   - Used for filtering, pagination, and search operations\n   - URL-encoded according to RFC 3986\n\n4. **Binary Data**\n   - Used for file uploads and downloads\n   - Transferred directly to/from S3 via pre-signed URLs\n\n### 4.3 Schema Evolution Approach\n\nThe SDK follows these principles for handling schema evolution:\n\n1. **Backward Compatibility**\n   - Optional fields use pointer types to distinguish between zero values and unset fields\n   - New fields are always added as optional\n   - Field removal follows a deprecation period\n\n2. **Versioning Strategy**\n   - The SDK uses semantic versioning (SemVer)\n   - Breaking changes are limited to major version increments\n   - API endpoints are versioned separately from the SDK\n\n3. **Extensibility Patterns**\n   - Functional options pattern for client configuration\n   - Maps for extensible metadata (`map[string]string`)\n   - Interface-based design for pluggable components (e.g., `TokenProvider`)\n\n4. **Field Type Flexibility**\n   -
